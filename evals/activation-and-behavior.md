# Проверка Активации И Поведения

Этот файл нужен для независимой проверки скилла. Не загружай его во время обычной онтологической сессии.

## Activation Cases

| Запрос | Должен сработать | Критерий |
| --- | --- | --- |
| "Давай разработаем бизнес-онтологию модуля привлечения" | Да | Агент начинает с поиска контекста и вопроса о границе. |
| "Продолжим онтологию, поменялся интерфейс между лидгеном и продажами" | Да | Агент сначала ищет существующие файлы и сравнивает с прежней моделью. |
| "Проверь, что в модели компании не определено и где дрифт" | Да | Агент входит в режим аудита/дрифта. |
| "Посчитай CAC за май" | Нет | Агент решает аналитическую задачу без загрузки скилла. |
| "Сделай BPMN процесса продаж" | Нет | Агент строит процессную схему, если пользователь не просит онтологию. |
| "Спроектируй ERD для CRM" | Нет | Агент не подменяет data schema бизнес-онтологией. |
| "Напиши регламент менеджера продаж" | Нет | Агент пишет документ, если нет задачи извлечь онтологические правила. |
| "Разложи модули и производственные системы для нового замысла" | Погранично | Агент уточняет, строится ли бизнес-онтология. |

## Behavior Cases

### Конфликт С Принятой Моделью

Вход: пользователь утверждает, что ранее принятый модуль теперь является инструментом.

Проходной результат:

- агент не соглашается автоматически;
- показывает, какие карточки и интерфейсы затронуты;
- предлагает варианты разрешения;
- после решения фиксирует дифф.

### Продолжение Без Контекста

Вход: пользователь просит продолжить существующую онтологию и изменить интерфейс.

Проходной результат:

- агент сначала ищет `business-ontology/`, журнал, границу, карту источников и дрифт;
- не начинает новую структуру;
- сравнивает новое утверждение с прежним;
- фиксирует изменение в журнале.

### Пустая Карточка

Вход: данных для карточки модуля мало.

Проходной результат:

- агент не оставляет пустые разделы;
- пишет `неизвестно`, `не применимо`, `кандидат` или `гипотеза`;
- важное неизвестное выносит в дрифт или открытые вопросы.

### Пользователь Просит Всю Компанию

Вход: пользователь просит сразу описать всю компанию.

Проходной результат:

- агент объясняет риск имитации полноты;
- рекомендует минимальную проверяемую границу;
- задает один вопрос о стартовой границе.

## Launch Gate

Скилл можно считать готовым к публикации, если агент проходит:

- все обязательные activation cases;
- минимум 3 из 4 behavior cases;
- не выполняет запись в файлы, когда пользователь просит только обсудить;
- не создает полную структуру до определения границы.
