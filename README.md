**Agent Skill для сборки, проверки и развития бизнес-онтологии — курируемой модели того, как реально работает модуль/компания: определения, состояния, решения, источники и дрифт.**

Не RDF/OWL и не схема БД. Это операционная сессия захвата и фиксации, которая оставляет после себя версионируемую модель в git, запрашиваемую агентами по `id`.

## Сценарии

### 1 — Первая сессия: снять модель модуля

```text
Ты    > Давай соберём онтологию модуля «Привлечение».
Скилл > Сначала намою каркас из артефактов (CRM-стадии, регламент, выгрузки),
        чтобы не спрашивать то, что уже зафиксировано.
        Вижу воронку из 12 стадий и 4 исхода. Граница: один модуль, не вся
        компания. Один вопрос: что считаем главным объектом —
        «Потенциальный участник» или «Сделка в воронке»?  (моя рекомендация —
        участник, сделка это CRM-оболочка). Подтвердишь — запишу карточку.
```

### 2 — Дрифт: поймать расхождение модель↔реальность

```text
Ты    > Лидген теперь не поставщик, а инструмент.
Скилл > Стоп: это конфликт с принятой моделью (карточка модуля + интерфейс).
        Варианты: переименовать проекцию / развести место и средство /
        пометить старое устаревшим. Рекомендую развести. Оформлю как дифф
        (было->стало, основание) и строку в CHANGELOG — решение за тобой.
```

### 3 — Подключить онтологию к рабочему проекту

```text
Ты    > Пропиши, чтобы агенты в проекте модуля читали онтологию.
Скилл > Добавлю строку в AGENTS.md проекта: «перед работой над модулем
        прочитай его онтологию, отвечай по ней, расхождение фиксируй как дрифт».
        Разместить на уровне проекта, мастер-папки или глобально — см. ai-ready.md.
```

## Что это

Скилл ведёт онтологическую сессию циклом **захвата и фиксации**: намывает каркас из существующих артефактов (mine-first), задаёт один сильный вопрос с рекомендованной формулировкой, и **сразу записывает** подтверждённое в карточку со статусом, источником и типизированными связями. git — источник истины модели; человек коммитит, агент предлагает.

Модель описывает **как реально работает сейчас** (as-is); регламентное «как должно» фиксируется только при расхождении, как зазор. Дрифт первокласен и имеет каденцию сверки. Стабильные непрозрачные `id` и закрытый словарь связей делают модель запрашиваемой агентами и склеиваемой оверлеями (дашборд, финмодель) по `id`.

## Философия

- Модель для людей **и** агентов: id + типизированные связи, а не только проза.
- As-is по умолчанию; «как должно» — помеченное исключение.
- Сначала намывай из артефактов, спрашивай только дырки.
- Подтвердил → записал → потом дальше. Не копить ответы в чате.
- git — истина модели; агент предлагает, человек фиксирует.
- Связь только из закрытого словаря; id непрозрачен и стабилен.
- Дрифт виден и сверяется по ритму, а не замалчивается.

## Структура

```text
business-ontology/
  SKILL.md                 # ядро поведения скилла
  references/
    structure.md           # три слоя, карта файлов, статусы, источники, drift-sweep
    templates.md           # карточки: понятие, модуль, произв-система, интерфейс, процесс, состояние, решение
    ai-ready.md            # стабильные id, закрытый список связей, проверка, проводка в AGENTS.md
    registry-spec.md       # контракт компиляции в граф: узлы/рёбра, англ. ключи, разложение интерфейса
    pressure-tests.md      # сценарии проверки поведения
  scripts/
    links_validate.py      # валидатор связей без зависимостей
  evals/
    activation-and-behavior.md
```

`SKILL.md` — только ядро; детали вынесены в `references/` и грузятся по необходимости (прогрессивное раскрытие).
