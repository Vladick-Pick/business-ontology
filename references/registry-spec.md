# Registry: Контракт Компилируемого Слоя

Markdown-карточки - источник истины. `registry` - производный машинный слой: граф, в который карточки компилируются **один-в-один** для запросов агентами, дашбордов, MCP и оверлеев по `id`. Его не пишут руками; он генерируется из карточек и проверяется.

Этот файл фиксирует контракт, чтобы «agent-queryable» было обязательством, а не обещанием, и чтобы потребители (например интерпретатор дашборда, финансовый моделлер) проектировались уже сейчас.

Читай, когда: проектируешь компиляцию в граф, MCP-слой или внешнего потребителя; вводишь новую связь; раскладываешь интерфейс.

## Принципы

- **Человеку - русский, машине - английский.** В markdown ключи и связи русские (читаемо для людей). В registry - канонические английские ключи (чисто для графа/LLM/тулов). Маппинг ниже; компиляция мостит, авторинг остаётся русским.
- **id непрозрачен и стабилен.** Узел графа = `id` карточки. `id` не выводится из имён и не меняется при переименовании (см. [ai-ready.md](ai-ready.md)). Человекочитаемое имя едет в `label`.
- **Связь - только из закрытого списка.** Новый тип ребра = решение + правка [ai-ready.md](ai-ready.md) и этого файла, не разовая выдумка.

## Схема Узла

```json
{
  "id": "ps-privlechenie",
  "type": "production-system",
  "subtype": null,
  "label": "Привлечение",
  "status": "accepted",
  "source": "решение рук. лидгена, 2026-06",
  "owner": "рук. лидгена",
  "last_reviewed": "2026-06-16",
  "next_audit": "2026-09-16",
  "attrs": {}
}
```

`type` (вид карточки): `concept | module | production-system | interface | process | state | decision`.
`subtype` - значение поля `тип` карточки понятия: `продукт | услуга | метрика | инструмент | роль | позиция | …` (или `null`).

## Схема Ребра

```json
{ "id": "ps-privlechenie::produces::vyh-kval-lid",
  "from": "ps-privlechenie", "to": "vyh-kval-lid",
  "type": "produces", "attrs": {} }
```

`id` ребра выводится детерминированно как `<from>::<type>::<to>` (это производное допустимо - ребро не переименовывают; запрет на производные id относится к УЗЛАМ).

## Ключи Фронтматтера: RU → EN

| markdown (ru) | registry (en) |
|---|---|
| `id` | `id` |
| `тип` | `subtype` |
| `статус` | `status` |
| `источник` | `source` |
| `владелец` | `owner` |
| `последняя-сверка` | `last_reviewed` |
| `следующий-аудит` | `next_audit` |
| `связи` | (рёбра, см. ниже) |
| `участники` | `participants` (узел интерфейса) |
| `исход` | `outcome` |
| `зазор` | `gap` (`as_is`/`as_should`/`why`) |

Значения статуса: `принято→accepted`, `кандидат→candidate`, `гипотеза→hypothesis`, `конфликт→conflict`, `устарело→deprecated`, `неизвестно→unknown`.

## Связи: RU → EN (закрытый список)

| связь (ru) | тип ребра (en) | домен → область |
|---|---|---|
| `производит` | `produces` | модуль/произв-система → выход |
| `потребляет` | `consumes` | модуль/произв-система → выход/инструмент |
| `поставляет-в` | `supplies_to` | роль-поставщик → роль-заказчик |
| `часть-чего` | `part_of` | модуль/система → модуль/система |
| `владеет` | `owns` | модуль → производственная-система/инструмент |
| `измеряется` | `measured_by` | процесс/система/выход → метрика |
| `источник-истины` | `source_of_truth` | метрика/факт/состояние → инструмент/система |
| `в-состоянии` | `in_state` | выход/процесс/сущность → состояние |
| `регулируется` | `governed_by` | система/роль/модуль → правило/решение |

Любой тип за пределами этой таблицы при компиляции - ошибка валидации (см. валидатор).

## Разложение Интерфейса (Гиперребро → Пары)

Интерфейс - это гиперребро (несколько участников + исход), но в графе он раскладывается детерминированно: **один узел `interface` + структурные рёбра + атрибуты**. Пример карточки:

```yaml
id: if-privlech-prodazhi      # непрозрачный, не из имён участников
тип: интерфейс
участники:
  поставщик: [role-privlech-postavshik]
  заказчик: [role-prodazhi-zakazchik]
  предмет: [vyh-kval-lid]
  критерий-качества: <…>
исход: <квалифицированный лид существует>
```

Раскладывается в:

```text
node  if-privlech-prodazhi  type=interface
      attrs: { quality_criterion, outcome }
edges:
  if-privlech-prodazhi  has_supplier  role-privlech-postavshik
  if-privlech-prodazhi  has_customer  role-prodazhi-zakazchik
  if-privlech-prodazhi  has_subject   vyh-kval-lid
  role-privlech-postavshik  supplies_to  role-prodazhi-zakazchik
      attrs: { interface: if-privlech-prodazhi, subject: vyh-kval-lid }
```

`has_supplier | has_customer | has_subject` - структурные рёбра registry (не бизнес-связи), поэтому в закрытый список авторинга не входят; они порождаются только разложением интерфейса.

## Валидация

Минимум, который проверяет `scripts/links_validate.py` (и обязан проверять любой компилятор):

- каждый `id` уникален; узловой `id` не содержит `--` и не выглядит производным от имён;
- каждый target в `связи` резолвится в существующий узловой `id` (нет висячих ссылок);
- тип каждой связи входит в закрытый список выше;
- у каждой карточки есть `id` и `статус`.

Привязка к проекту: если у модуля уже есть боевой `registry/<module>-ontology.json` (как в attraction), компиляция должна давать совместимый формат - сверяй ключи и типы рёбер с ним, не плоди второй несовместимый контракт.
