# AI-Ready: Стабильные ID, Связи, Подключение

Этот файл делает онтологию запрашиваемой агентом, а не только читаемой как текст. Маленький слой поверх обычных карточек: стабильные `id`, короткий закрытый список связей и одна инструкция для рабочего проекта.

Читай, когда: создаёшь/правишь карточку, ставишь связь между сущностями, или подключаешь онтологию к рабочему репозиторию.

## Стабильные ID

У каждой карточки во фронтматтере есть `id`.

- `id` стабилен: не меняется при переименовании сущности. Человекочитаемое имя может меняться, `id` - нет.
- `id` уникален в пределах онтологии модуля.
- Стиль: короткий kebab-case, понятный человеку (`lead-quality`, `ps-privlechenie`, `crm`).
- Все ссылки между карточками идут по `id`, а не по имени. Это позволяет внешнему потребителю (например финансовому оверлею) приклеиться к карточке по её `id` и не зависеть от формулировок.

## Закрытый Список Связей

Связь в блоке `связи` карточки берётся только из этого списка. Если нужной связи нет - это сигнал, что список надо расширить осознанно (как решение, с записью в CHANGELOG), а не выдумать связь на ходу.

| Связь | От кого | К кому |
|---|---|---|
| `производит` | модуль, производственная система | выход/продукт |
| `потребляет` | модуль, производственная система | выход/продукт, инструмент |
| `поставляет-в` | роль-поставщик в интерфейсе | роль-заказчик в интерфейсе |
| `часть-чего` | модуль, производственная система | модуль, производственная система |
| `измеряется` | процесс, система, выход | метрика |
| `источник-истины` | метрика, факт, состояние | инструмент/система, где живёт факт |
| `в-состоянии` | выход, процесс, сущность | состояние |
| `регулируется` | система, роль, модуль | правило/полномочие |

Список намеренно короткий. Растёт только когда реально упёрся - и тогда новая связь добавляется сюда первой, а потом используется.

## Проверка Висячих Ссылок (вручную)

Перед фиксацией правки пройди по `связи` карточки: каждый `id` в значениях должен резолвиться в существующую карточку с таким `id`. Висячая ссылка (id, которого нет) - это либо опечатка, либо недостающая карточка-кандидат. Не оставляй молча.

Автоматическая проверка (валидатор, impact radius, запрос по графу) - это зрелый слой `registry/`, его в MVP нет. Пока это ручная дисциплина. Не утверждай «проверил связи», если фактически не прошёл по ним.

## Подключение К Рабочему Проекту

Чтобы онтология подхватывалась во время работы (вайбкодинг рядом с модулем), добавь в `AGENTS.md` (или `CLAUDE.md`) рабочего репозитория строку-инструкцию:

```markdown
Онтология этого модуля живёт в <repo/путь к business-ontology>.
Перед работой над модулем прочитай релевантные карточки и отвечай «как сейчас» по модели.
Если видишь расхождение модели с реальностью - не проходи мимо: предложи правку как дифф
(было->стало, основание), добавь строку в CHANGELOG.md под именем человека и дай человеку
зафиксировать. Сам коммит за себя не делай.
```

### Где разместить правило

- **На уровне проекта** (по умолчанию): строка выше - в `AGENTS.md`/`CLAUDE.md` рабочего репозитория модуля. Самый явный и локальный вариант.
- **На уровне мастер-папки**: если все проекты лежат под одной над-папкой, положи `AGENTS.md` в над-папку с общим правилом «для каждого модуля читай его онтологию из <карта модуль->репо> и используй скилл business-ontology». Один файл покрывает все проекты.
- **Глобально**: правило в `~/.claude/CLAUDE.md` подхватывается в любой сессии. Удобно, но шумит там, где онтология не нужна; бери, если онтологическая работа у тебя везде.

Как агент достаёт содержимое - git submodule, sibling-клон или `git pull` по требованию (деталь настройки). Для MVP достаточно, чтобы репо онтологии было доступно агенту локально.

Граница честности: агент ловит **противоречие в контакте** (в сессии всплыло то, что расходится с моделью), а не «сканирует реальность». Магического авто-детекта дрифта нет - ловим то, что всплыло в работе.

## Что Это Готовит На Будущее (не в MVP)

`id` + `связи` - это семя под целевой query-слой. Когда (и если) появится потребность отдавать онтологию сервисам по сети, эти же `id` и типы связей компилируются один-в-один в машинную форму (граф/гиперребро) за MCP-сервером, а потребители (интерпретатор дашборда, финансовый моделлер) читают и клеят оверлеи по `id`. Сейчас этого слоя нет и строить его не нужно - но писать `id` и связи стоит уже сейчас, чтобы потом не переразмечать.
